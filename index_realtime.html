GhostSched (Custom)
POWER-AWARE · THERMAL · EMA PREDICTION
  • DVFS: scales V and f based on real-time load
  • EMA predictor: anticipates next burst, pre-scales
  • Thermal migration: moves tasks away from hot cores
  • big.LITTLE placement: HIGH priority → P-core, LOW → E-core
  • Energy model: P = C × V² × f
Round Robin
TIME-SHARING · QUANTUM=4ms · NO ENERGY AWARENESS
  • Fixed time quantum: each task gets 4ms CPU slice
  • No priority distinction — treats all tasks equally
  • No DVFS: CPU runs at constant max frequency
  • High fairness, but poor energy efficiency
  • Context switch overhead: O(n) queue traversal
Shortest Job First (SJF)
NON-PREEMPTIVE · OPTIMAL WAIT · STARVATION RISK
  • Always runs shortest burst_time task next
  • Provably minimizes average waiting time
  • Risk: long tasks may starve (never get CPU)
  • Requires burst time prediction in practice
  • No thermal or energy management
Priority Scheduling
PREEMPTIVE · PRIORITY QUEUE · AGING SUPPORT
  • Tasks sorted by priority (0=HIGH, 1=MID, 2=LOW)
  • High-priority tasks always preempt lower ones
  • Aging: priority boosts over time to prevent starvation
  • Good for real-time and interrupt-driven systems
  • No DVFS, no thermal awareness

REAL-WORLD APPLICATIONS
📱 Mobile Processors
ARM big.LITTLE · APPLE M-SERIES · QUALCOMM SNAPDRAGON
  • GhostSched mirrors the big.LITTLE heterogeneous core architecture used in modern ARM SoCs
  • Thermal-aware scheduling prevents throttling during sustained gaming or video encoding
  • DVFS is standard on all modern smartphone SoCs — Snapdragon 8 Gen 3, Exynos 2400
  • EMA-based workload prediction helps pre-scale before UI gesture response
  • Battery life directly determined by scheduling energy efficiency
🔌 Embedded Systems
RTOS · FREERTOS · ZEPHYR · BARE-METAL MCU
  • Priority scheduling is the foundation of FreeRTOS and Zephyr RTOS task management
  • Lightweight EMA (O(1) memory, O(1) compute) is feasible even on Cortex-M0+ at 48 MHz
  • Energy constraints critical: sensor nodes must run months on coin-cell batteries
  • Thermal modeling important in automotive and industrial embedded controllers
  • Deterministic scheduling deadlines (hard real-time) require priority-based preemption
🌐 IoT Devices
EDGE COMPUTING · SENSOR NODES · SMART HOME
  • IoT devices often have heterogeneous compute: main MCU + radio coprocessor + sensor DSP
  • Task migration between compute units mirrors our thermal migration in GhostSched
  • EMA prediction enables duty-cycle optimization — sleep between predicted idle periods
  • Round-Robin fairness needed when multiple sensors share same MCU timeslice
  • Power envelope: IoT nodes often restricted to <10mW continuous — DVFS essential
🖥 Data Centers & Cloud
KUBERNETES · HYPERVISORS · NUMA SYSTEMS
  • Thermal-aware task placement reduces cooling costs (PUE optimization)
  • DVFS at rack level: Intel RAPL, AMD EPYC frequency scaling
  • Workload prediction enables proactive VM migration before thermal events
  • Fairness index critical in multi-tenant cloud environments (SLA guarantees)
  • Google Borg, Linux CFS all use weighted fair-share scheduling with priorities
LIMITATIONS OF THIS SIMULATION
⚠ Simulation Constraints
NOT A REAL OS KERNEL
  • This is a simulation — not running in a real OS kernel or hypervisor
  • All task burst times, priorities and arrival times are synthetically generated
  • Temperature model is simplified — real thermal behavior is far more complex
  • No actual hardware registers, MSRs, or ACPI P-states are manipulated
  • Context switch overhead, cache effects, and memory latency are not modeled
  • EMA predictor uses idealized input — real workloads are noisier and spikier
⚡ Energy Model Limits
SIMPLIFIED P = C·V²·f
  • Static power (leakage current) not included — significant at small process nodes
  • Memory subsystem energy (DRAM, cache) excluded from calculation
  • Voltage domains are simplified — real chips have 5–15 independent DVFS domains
  • I/O power (PCIe, USB, WiFi) not simulated — often 30-50% of total SoC power
  • Process variation and silicon lottery effects not modeled
